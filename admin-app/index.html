Good Food — Admin App

The web control room for your Good Food platform: oversee users, restaurants, drivers, orders, and settings from a browser—without opening a code editor.

This overview is for non-technical stakeholders and developers alike. You will see what the dashboard is for, who uses it, and how it stays in sync with the same API that powers the mobile apps.

Easy to launch Web dashboard Self-hosted REST API

Overview (Presentation)

The Good Food admin app is the back-office product in the suite. Mobile apps serve customers, restaurants, and drivers; the admin experience is where platform operators create and maintain the data those apps rely on.

It is a React web application that runs in modern browsers. After a secure sign-in, staff can browse structured lists, open detail pages, create or edit records, and consult reporting views—similar to other SaaS consoles you may already know.

All sensitive operations are enforced by your backend: the browser only shows what the API allows for the signed-in role.

One API, many surfaces

Customers, restaurants, drivers, and admins all talk to the same Good Food REST API. Changing a restaurant record here can change what the customer app displays; cancelling an order may notify the driver app—the admin panel is the holistic view.

Admin App Presentation

The layout is intentionally desktop-first: wide tables, filters, and multi-column dashboards that would feel cramped on a phone. Many screens still adapt when you resize the window or use a tablet in landscape.

Day-to-day work often looks like: search for an entity (a restaurant, an order, a user), open it, adjust fields, save, and move on. Repetitive list/detail/new patterns are shared across modules so training one area helps with the others.

For internal testing, the login experience may offer quick ways to fill sample roles—treat those as development aids, not production shortcuts.

Main Function

Give trusted staff a single web hub to configure the platform, monitor operations, and support users—always backed by server-side permissions.

Conceptually, you start on an authentication screen, then land on a home or dashboard view, then use a sidebar (or equivalent navigation) to jump between major domains.

Under the hood, many routes are generated from a central catalogue of “business entities” so adding a new resource does not always mean hand-crafting three new pages.

Supported Platforms

  • Modern web browsers on desktop (primary experience)
  • Tablet or laptop browsers (responsive layout)

There is no separate native store app: you deploy the built static assets to the host of your choice and users visit a URL.
