Good Food — Restaurant App

The mobile workspace for restaurant teams: see new orders, update the menu, and understand how the service is performing—without digging through code.

This overview is written for non-technical readers (owners, managers, buyers) as well as developers. You will understand what the app is for, how it behaves for staff, and how it connects to the same backend as the customer and driver apps.

Easy to launch Android & iOS ready Self-hosted REST API

Overview (Presentation)

The Good Food restaurant app is the operations companion in the suite. While the customer app is where people place orders, the restaurant app is where your team receives those orders, prepares them, and keeps the menu accurate.

Like the rest of the stack, it talks to your Good Food REST API and database. You host the backend; the app only displays and updates what the server authorizes for a signed-in restaurant account.

Nothing in this product locks you into a proprietary cloud: API URL, branding, and behaviour are yours to configure.

Same backend as customer and driver

Orders created on the customer app appear here when your API routes them to the right restaurant. Status changes you make here are what customers and drivers see next—one coherent pipeline, three tailored apps.

Restaurant App Presentation

The interface is meant for busy teams: large touch targets, clear statuses, and flows that match real service (new order → preparing → ready → handed off). You do not need to be a developer to use it day to day.

Typical tasks: review the live order list, mark steps as done, adjust items or availability on the menu, glance at sales or activity, read feedback, and change preferences (notifications, language, and similar options).

A demo mode is available for presentations: it can preload sample login details so stakeholders can explore without touching production data. Turn it off before going live.

Main Function

Give restaurant staff one reliable mobile hub: stay on top of incoming orders, keep the menu truthful, and see whether service is running smoothly—all synced through your API.

After sign in, users land in a guided structure (dashboard, orders, menu, analytics, reports, and more) similar to how other modern ops apps work: a main menu, then detail screens for each task.

The app is designed to stay responsive: data can refresh in the background so lists do not freeze while the kitchen is in a rush.

Supported Platforms

  • Android
  • iOS

One shared React Native / Expo codebase serves both stores, so updates and fixes roll out consistently.

Package identity in the project is aligned under the Good Food restaurant bundle name for both platforms (see app metadata in the repository if you publish to stores).

Technically, this is a standard React Native + Expo client: screens, reusable UI pieces, and hooks that call your HTTP API. Business rules and persistent data stay on the server; the phone is the control surface.

To point the app at your environment, you set the API base URL and related options in a single configuration file at the root of the restaurant project (see Getting Started).
