Good Food — my-backend

The Node.js API and database layer every Good Food client talks to: auth, catalog, orders, drivers, settings, uploads, and more.

This overview is for non-technical readers (buyers, PMs) and engineers alike: what the backend does, how it connects to mobile and web apps, and why it must run before you test any front-end.

Express.js + MongoDB REST under /api Socket.io ready

Overview (Presentation)

my-backend is the reference Good Food server for the suite. The customer, restaurant, driver, and admin apps are thin clients: they send HTTP requests (and may use real-time channels) while this service stores users, menus, orders, and configuration in MongoDB.

You host it yourself (laptop, VPS, cloud). Mobile apps point to something like http://YOUR_IP:5000/api; the admin dashboard uses the same base URL. No third-party Good Food cloud is required.

Start the API before the apps

If the backend is stopped, every client shows errors, empty lists, or failed login—even when the front-end install is perfect. For local work, run npm run dev (or npm start) in my-backend first, then launch Expo or the admin site.

What the API provides

In plain terms: accounts and roles (customer, restaurant staff, driver, admin), catalog (restaurants, categories, products, menus), cart and checkout flows, orders and related updates, drivers, reviews, settings and languages, file uploads for images, and generic CRUD routes used heavily by the admin dashboard.

Authentication uses JWT; protected routes sit under /api after the auth middleware. Cookies and CORS are configured for browser clients (admin).

Main Function

Be the single source of truth for data and permissions: every app reads and writes through the same REST surface and database.

The process is standard for a food platform: sign in → receive a token → call endpoints with that identity → server validates role and returns or updates documents in MongoDB.

Supported Platforms

  • Node.js (LTS recommended)
  • MongoDB (local or Atlas)

Default HTTP port is 5000 when PORT is unset. Match that (or your chosen port) in every client’s API base URL.
